## What is NameForge?

NameForge is a name availability intelligence platform built by Loman Cavendish. A user enters a proposed company or product name and receives a scored report covering domain availability, company-registry conflicts, trademark conflicts, and similarity risks across selected jurisdictions (initially France, United Kingdom, European Union). Evidence is captured per finding and the report is exportable as Markdown and JSON. NameForge provides structured availability intelligence — it does not provide legal advice.

## Operating Protocol

1. Propose first, then implement. Wait for approval before making changes.
2. One commit per completed task with a detailed message.
3. Developer performs git push manually.
4. Run tests after each task and report the outcome.
5. Use British English throughout.
6. Work is tracked in Jira project NMF on lomancavendish.atlassian.net. Move tickets through the workflow as work progresses.

## Architecture

- Three-service Docker Compose stack (nmf-db, nmf-api, nmf-web)
- Clean architecture with adapter pattern for every external registry (domain, company, trademark)
- All connectors implement a common `AvailabilityConnector<TRequest, TResult>` interface
- Jurisdiction logic lives in adapters, not in core services
- Similarity engine is deterministic and explainable (Levenshtein + Jaro-Winkler + normalisation)
- Risk scoring is rules-based and every component of the score is attributable to a source finding
- Evidence is stored for every finding (source name, URL, timestamp, raw reference, summary)
- Reports are generated from stored evidence — never regenerated by re-querying external services
- Every report carries the "not legal advice" disclaimer

## Key Rules

- Never commit `.env` files, credentials, API keys or secrets
- Never hard-code jurisdiction logic into core services — it belongs in adapters
- Every external call must be cached by (connector, normalised input) to avoid duplicate API spend
- Every finding must carry an evidence record before it can contribute to a risk score
- Risk scoring must remain explainable — if you cannot state which finding contributed which points, the score is broken
- Validate at system boundaries only — trust internal code and framework guarantees
- No over-engineering for hypothetical future requirements
- Follow the propose-then-implement pattern for every task
- Every report must include the disclaimer: "This report provides automated availability intelligence only. It is not legal advice, does not guarantee registrability, and should not replace professional trademark or legal review."
